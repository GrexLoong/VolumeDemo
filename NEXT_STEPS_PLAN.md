# 阶段 5 详细工作规划：安卓真机投产与应用生命周期完善

经过核心组件调参、窗口拖动稳定性修复以及 UI 交互级的全面升级（对标 iOS 沉浸式动效），目前纯桌面环境下的 `Mock` 模式表现已具备高保真度。接下来的重点将从“桌面端视觉调试”转移到“Android 端深度集成与投产准备”。

以下是下一步详细的推进路线图与工作规划：

## 一、 安卓环境真机链路验证与调优 (Mobile Integration)

### 1。 Buildozer 打包环境收尾
- **工作项：** 细化和检查 `buildozer.spec` 配置。
  - 确认 `android.permissions = RECORD_AUDIO` 权限声明明确。
  - 设置全屏/沉浸式状态栏配置（取消 ActionBar 显示），匹配已有的黑化沉浸式 UI。
  - 增加对应 Android NDK / SDK Target 匹配（建议 target_sdk 至少 33）。
- **验证标准：** 成功编译产出 `app-debug.apk`。

### 2。 麦克风音频流 (MicrophoneSource) 真实性能适配
- **工作项：** 真实真机环境下拾音及绘制。
  - 由于桌面端是用加噪 Mock 信号模拟动态，而在真机跑 `MicrophoneSource` 时可能面临底噪过高或者增益收音不灵敏的问题。
  - 需针对 `MAX_EXPECTED_RMS` 参数以及 `AudioRecord` 的真实拾音特性引入 **归一化校准或动态增益算法 (AGC)**，避免正常说话时波形过小或轻易触顶（爆音裁切）。
- **验证标准：** 在真机环境下，对着麦克风正常说话和呼气，波形效果是否同桌面端 Mock 模式一致。

## 二、 生命周期管理及边缘 Case 修复 (App Lifecycle)

### 1。 熄屏与后台机制 (Pause & Resume)
- **工作项：** 手机应用切后台与熄屏时，Kivy 默认暂停绘制但后台音频可能会报异常。
  - 拦截 Kivy 的 `on_pause()` 和 `on_resume()` 事件。
  - 如果允许后台录制：将 UI 休眠、保留底层线程写入。
  - 否则：在退入后台时自动停止录音或挂起音频采样与绘制，待重新唤醒时恢复。
- **验证标准：** 在录音中途按下 Home 键返回桌面再切回，App 不能崩溃，录音计时应能继续或正常停止。

### 2。 屏幕旋转与重力感应重绘
- **工作项：** 目前波形是以视口变量为重绘锚点，但并未针对宽高比互换做测试。
  - 分析横竖屏切换（或强行锁定竖屏）。iPhone 录音备忘录通常锁定竖屏。
  - 在 `buildozer.spec` 配置 `orientation = portrait`，避免旋转带来的重新初始化。

## 三、 文件流数据分离（未来的核心业务架构）

- **工作项：** 目前的波形仅做了“视口滚动和抛弃”（`deque` 容量固定，走出的丢弃），还没有加入存储落盘逻辑。
  - 准备解耦：一方面音频缓冲推送给前端做波形渲染，另一方面原始 PCM / WAV 流应开始逐步写入到本地文件，实现“录音 - 波形图 - 存文件” 的完整闭环。
- **验证标准：** 设计出流式回写本地音频文件结构，并在停止录制后生成可播放的 `.wav` 文件。
